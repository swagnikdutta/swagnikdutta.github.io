Day 1

Toy DNS resolver

Spent most of my daytime reading the RFC pages. I took time to understand the structure of the DNS message packet in general and each of its sections and that of Resource Records.


    +---------------------+
    |        Header       |
    +---------------------+
    |       Question      | the question for the name server
    +---------------------+
    |        Answer       | RRs answering the question
    +---------------------+
    |      Authority      | RRs pointing toward an authority
    +---------------------+
    |      Additional     | RRs holding additional information
    +---------------------+

Creating the packet is essentially the first step. Which is followed by serialization of it. That’s what I have learnt from implementing ping few weeks ago. Except that, ping was much simpler.

On the coding part I didn’t achieve much. I created the types representing the Message and its sections with documentation for each of the fields so that I can avoid switching from Goland to RFC pages frequently.

Volunteered to use TCP implementation guide by Julia Evans

At night I volunteered to use Julia Evan’s guide to implement TCP in a weekend, which is in alpha right now.

Doing this exercise would mean using the guide to implement TCP in a language of my choice(Go) and report my findings in the process. I feel good about this exercise and it aligns with my goal at RC as well.

Spent sometime reading the code at a high level. I focused on the TCP checksum calculation part which seemed strange. I referred to the TCP RFC pages to learn that checksum calculation in TCP is somewhat differnet. It considers certain informations from the IP header as well which it calls pseudo-header.

Generally, if packet A encapsulates packet B, pakcet A has to consider the entirety of packet B while calculating checksum. But here it seemed that, packet B was interested in certain information from packet A’s header.
